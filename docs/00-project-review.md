# 《牛马大逃杀》综合项目评审

评审日期：2026-07-12
评审对象：Web/Android 垂直切片，`package.json` 版本 `0.3.0`
评审方式：仓库静态审阅、数据表核对、资源检查、生产构建验证

## 1. 结论

《牛马大逃杀》已经不是单纯玩法原型，而是一款内容量可观、主题表达鲜明、具备完整局内闭环和轻量局外系统的可玩垂直切片。它最有价值的部分不是“幸存者-like + 大逃杀”的表层组合，而是把办公室黑色幽默贯穿到规则、文案、UI 文书、怪物、技能和结算中的统一表达。

当前项目适合进入“收敛与产品化”阶段，不适合继续无边界扩内容。影响质量的首要问题已经从“缺内容”转为“版本漂移、系统边界不清、内容池污染、验证能力不足”。建议先完成一个稳定的 v0.4 基线，再决定是否扩展角色、人设和新地图。

综合评级：

| 维度 | 评级 | 判断 |
|---|---:|---|
| 核心概念 | A | 一句话卖点清晰，题材与机制结合度高 |
| 核心循环 | B+ | 试用期 PVE → 转正 PVP → 缩圈 → 老板终局的节奏有辨识度 |
| 内容与构筑 | B | 内容丰富，但池子过宽且人设系统未真正闭环 |
| 数值与节奏 | B- | 已有较多保护机制，但缺少可重复仿真和遥测支撑 |
| 美术与 UI | B+ | 低成本形成统一风格；角色与场景资产深度仍不足 |
| 技术架构 | B | 引擎/UI 分层合理；核心文件过大、测试与 schema 缺失 |
| 在线与运营 | C+ | 云存档、榜单、每日挑战骨架已具备，可信结算和奖励闭环不足 |
| 发布准备度 | C+ | 能构建和运行，但距离可持续发布仍缺测试、性能预算、隐私与内容合规检查 |

## 2. 已验证事实

- 技术栈：React 19、Vite 6、原生 Canvas 2D、纯 JS 游戏引擎、Capacitor Android、可选 Supabase。
- 生产构建成功：117 个模块；构建目录约 440 KB；主 JS 416.13 KB，gzip 149.16 KB；CSS 16.51 KB，gzip 4.39 KB。
- `npm audit`：0 个已知漏洞。
- 当前主武器池 12 把；代码总武器定义 21 把；传说定义 12 把；实际配方 8 条。
- 技能 37 条，其中 17 条带人设标签；副武器 17 件，其中 5 件带人设标签；主动技能 12 个，其中 4 个带人设标签。
- 技术模组 13 个、精英/小 Boss 10 个、杂鱼 29 个、消耗品 20 个、永久升级 10 项、成就 10 项。
- 模式：经典大逃杀、每日挑战、无尽模式；经典模式支持 0—6 个月试用期。
- 当前地图为单张 2200×2200 程序化办公室，逻辑视口 640×360。

## 3. 核心优势

### 3.1 主题不是皮肤，而是系统语言

“优化、试用期、转正、裁员红线、月度考核、老板画饼、HR、N+1”均对应真实规则。开始页像红头文件，升级页像晋升审批，失败页是离职证明，胜利页是优秀员工证书。主题词、交互反馈和玩法动作同构，记忆点远强于单纯更换角色皮肤。

### 3.2 对局结构具有原创辨识度

经典局不是从第一秒开始 20 人乱斗，而是可选试用期 PVE 发育，再让 19 名同事带追赶成长空降，随后进入缩圈 PVP，最终老板必须被击败。该结构同时解决新手缓冲、构筑起步和终局叙事问题，是产品最值得保护的机制资产。

### 3.3 数据驱动程度良好

武器、技能、怪物、模组、道具、事件、商店、成就等均拆到 `src/game/data/`。新增内容通常不需要修改 React 层。引擎通过 bridge 向 React 发事件和版本通知，HUD 约 10 Hz 刷新，Canvas 保持逐帧渲染，方向正确。

### 3.4 输入与平台覆盖务实

键鼠支持手动、自动开火、全托管三档；移动端使用动态摇杆、自动瞄准和触屏按钮。Canvas 固定逻辑分辨率配合整数像素风，适合低成本覆盖桌面浏览器和 Android WebView。

### 3.5 已具备轻运营骨架

本地存档、金币、永久升级、成就、图鉴、云存档、排行榜、每日挑战和离线待上传队列均已存在。虽然尚不完整，但产品从“单局玩具”升级到“可持续回访”的关键接口已经搭好。

## 4. 主要问题与风险

### P0：设计版本漂移，文档与代码存在多套口径

代码注释混合 v2.0、v4.1、v4.2，`package.json` 却为 0.3.0；README 宣称“12 武器 + 6 传说”，实际定义为 21 普通、12 传说、8 配方，主池才是 12。旧文档又引用不同公式和行号。若不先建立版本基线，后续平衡、QA 和发行文案会持续互相矛盾。

建议：以产品版本 `0.4.0` 建立唯一 changelog；数据对象增加 `status: live|experimental|legacy`；文档只描述 live 内容，实验内容单列。

### P0：人设专属内容未被人设系统约束

技能、副武器、主动技能和觉醒带有 `persona: optimizer|slacker`，但玩家没有明确的人设选择，`buildDraftPool()` 也未按 persona 过滤。因此两个设计成完整流派的专属内容会混入公共池，造成：

- 卡池从 20 条通用技能膨胀到 37 条，目标构筑出现率下降。
- 玩家可能同时拿到两个互斥人设的内容，破坏主题和强度预期。
- 专属觉醒缺少可理解的入口，玩家无法规划。

建议二选一：短期隐藏全部 persona 内容，保留稳定公共池；或正式加入开局人设选择，并过滤技能、副武器、主动技能、觉醒和 UI。不要维持中间状态。

### P0：每日挑战规则展示多于实际生效

每日挑战 UI 显示 `enemySpeedMul`、`goldMul`、`bossTime` 和通关奖励，但当前 `startGame('daily')` 只应用 `weaponPool`；其余 modifiers 和 `reward_gold` 没有进入核心结算，`daily_claims` 也没有客户端领取流程。玩家会看到规则却玩不到规则，属于信任风险。

建议：在 `newGame` 生成后统一执行 `applyRunRules(rules)`；奖励由服务端 RPC 或受约束数据库函数发放，客户端不能直接改金币。

### P0：缺少自动测试与确定性仿真

当前无 `test`、`lint`、`typecheck` 脚本。核心规则集中在 3518 行 `core.js`，随机数直接使用 `Math.random()`，无法复现同一局。构建能证明语法和打包成立，不能证明伤害、掉落、波次、存档与模式规则正确。

建议优先加入：

1. 可注入 seed 的 RNG。
2. 纯函数单测：配方、升级需求、商店、计分、反作弊、伤害乘区。
3. headless 仿真：10 分钟经典局、20 波无尽、每日挑战规则。
4. Playwright 冒烟：菜单 → 开局 → 暂停 → 结算 → 存档。

### P1：核心模块过大且运行时 schema 隐式

`core.js` 同时负责实体模型、战斗、AI、波次、Boss、掉落、构筑、元进度和在线提交。单位对象通过不断追加字段表达所有实体类型，字段契约靠注释维护。新增机制容易产生未初始化字段、跨系统耦合和回归。

建议按领域拆成 `state/`, `combat/`, `systems/`, `ai/`, `content-runtime/`；先拆纯函数和调度器，不需要重写 ECS。为 `GameState`、`Unit`、`Projectile` 建 JSDoc typedef 或 TypeScript 类型。

### P1：内容数量超过表现层承载能力

大量不同怪物共用少数精灵；许多技能只有文字和数值，没有独立图标、动画或声音。玩家难以在高密度战斗中区分“为什么受控、为什么掉血、谁是优先目标”。这会把丰富内容压缩成视觉噪声。

建议为威胁机制建立统一视觉语法：冲刺、远程、光环、复活、护盾、控制、易伤分别拥有固定轮廓、预警色和地面符号。先提升辨识度，再增加怪物数量。

### P1：地图是随机摆放，不是可验证的关卡

72 张桌子与 48 个装饰随机生成，只有中心安全留白和简单间距检查。没有导航网格、出生区语义、战斗房间、资源风险点或遮挡密度预算。程序生成可复玩，但也会出现视线碎裂、拾取物卡位、终圈家具过密等不可控情况。

建议建立 32×32 逻辑网格和预制区块：开放工区、会议区、茶水间、服务器间、老板办公室；随机化区块布局而不是逐件随机家具。

### P1：在线成绩可信度有限

分数和 `suspicious` 均由客户端计算，数据库允许用户插入自己的成绩。`run_hash` 不是签名，不能阻止修改客户端或直接调用 API。排行榜适合作为轻社交展示，不应承诺竞技公平。

建议在文案中定位为“轻榜单”；若要竞技化，使用 Edge Function 校验事件摘要、版本、挑战 seed 和速率上限，并限制每个玩家/模式的写入频率。

### P1：IP、商标与内容合规需在发行前确认

游戏直接使用多个 AI 品牌名称与职场讽刺文案。作为内部原型问题不大，但商业发行前需要评估商标使用、品牌视觉相似性、应用商店内容政策、隐私政策以及 Supabase 数据说明。美术应避免复刻品牌 logo，以文字和原创色彩/弹道表达为主。

### P2：包体与加载可继续优化

主 JS gzip 149 KB 对当前项目可接受，但 Supabase 被打入首屏包。在线功能可动态导入；覆盖界面也可按需拆包。高保真资源异步加载已有回退机制，是良好基础。

### P2：可访问性与小屏信息密度不足

大量信息依赖红/绿/紫颜色和 7—11 px 字号；移动端高密度 HUD 容易遮挡。需要色盲冗余符号、最小字号基线、可调 HUD 缩放、震屏开关和低特效模式。

## 5. 数值与体验判断

### 5.1 经典局时长跨度过大

0 个月试用期直接进入大逃杀；6 个月试用期理论时长合计 510 秒，之后才开始正式缩圈计时。虽然长试用期会把缩圈节点前移，但完整局仍可能显著超过 15 分钟。菜单把 0—6 个月都作为并列选择，会稀释标准体验和榜单可比性。

建议：对外只提供“快速 2 月”“标准 3 月”“硬核免试”三个预设；4—6 月放入自定义/练习，不计经典榜。

### 5.2 构筑池宽度与升级次数不匹配

公共池已经包含 20 技能、17 副武器、12 主动技能，加上人设内容后更宽。即使有连续三次未出装备的软保底，玩家仍难围绕配方规划。当前升级曲线到 Lv.15 累计需 1815 经验，构筑完成度高度依赖试用期长度和随机掉落。

建议：按局内阶段分池；Lv.2—5 保底提供副武器/主动入口，Lv.6 后开放高阶技能；重抽资源应可预测获得。

### 5.3 Boss 是正确的终局门槛，但需要遥测校准

Boss 在正式战斗 420 秒或剩余牛马不超过 3 时出现，血量随时间、等级、模组、副武器、蒸馏和传说调整，并会反向蒸馏玩家能力。这是强主题、强机制的设计。风险在于战力估算没有覆盖暴击、实际武器模式和部分触发链；半血狂暴是否能稳定持续 8—15 秒需要真实数据。

建议记录 Boss 出场时间、玩家 DPS、Boss 生存时间、狂暴持续时间、死亡来源和构筑快照，以 P50/P90 而不是单局体感调参。

## 6. 推荐路线图

### v0.4：稳定基线（优先，1—2 周）

- 冻结新内容，统一版本、术语与 live 内容清单。
- 决定 persona 内容隐藏或正式选择，并完成全池过滤。
- 让每日挑战全部 modifiers 与奖励真实生效。
- 引入 seed RNG、核心单测、3 条 headless 仿真和基础 CI。
- 修复明显文案/注释漂移，补存档迁移版本。
- 加入性能、可访问性和移动端 HUD 设置。

验收：三种模式可重复构建；每日挑战规则与 UI 一致；经典标准局连续 20 次仿真无软锁；无 P0 数据/存档问题。

### v0.5：关卡与表现升级（2—4 周）

- 用区块化办公室替代逐件随机家具。
- 为 8 类威胁建立视觉语法与音效提示。
- 补齐角色、Boss、精英与主流怪物动画；统一原生高保真和程序精灵尺度。
- 加入局内统计与匿名遥测开关，形成平衡看板。

验收：关键威胁盲测识别率大于 80%；低端 Android 稳定目标帧率；终圈不生成不可通行结构。

### v0.6：产品化与运营（视目标决定）

- 正式角色/人设选择，最多先发布 3 个完整流派。
- 每日挑战 seed 确定性、服务端奖励、赛季榜单。
- 新地图或办公室部门变体、任务/解锁线、首次用户体验。
- 完成商标、隐私、内容分级和商店页素材检查。

## 7. 不建议现在做的事

- 不继续平铺更多武器、怪物和技能；先让现有内容可读、可控、可测。
- 不重写为大型 ECS 或换引擎；当前架构足以支撑下一阶段，渐进拆分收益更高。
- 不把当前排行榜包装成强竞技系统。
- 不同时维护 0—6 月全部参数为独立平衡模式；应先确立一个标准局。
